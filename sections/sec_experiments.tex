\section{Results}


\subsection{Communication Benchmarks}


Communication performance is a key scalability constraint in MAS architectures. The benchmarks in this section quantify latency, throughput, reliability, and resource overhead for representative messaging patterns across multiple protocol families.


\subsubsection{Protocol options}
Inter-agent communication enables coordination, knowledge sharing, and joint decision making in multi-agent systems (MAS). As agent populations and message rates increase, protocol semantics (synchronous request/response versus brokered publish--subscribe), serialization formats, and transport behavior determine both performance and robustness. Table~\ref{tab:comm-protocol-options} summarizes the protocol options and variants evaluated in the benchmark suite.

\begin{table}[H]
\centering

\label{tab:comm-protocol-options}
\begin{tabular}{l|l|l|l}
\toprule
\textbf{Protocol} & \textbf{Interaction model} & \textbf{Serialization} & \textbf{Variants benchmarked} \\
\midrule
REST (HTTP) & Request/response & JSON over HTTP/1.1/2 & \texttt{http1}, \texttt{http2} \\
gRPC & RPC (unary/streaming) & Protobuf over HTTP/2 & \texttt{unary}, \texttt{streaming} \\

\paragraph{REST (HTTP).}
RESTful communication exposes agent interactions as stateless HTTP requests (typically JSON payloads)~\cite{Fielding2000REST,RFC9110}. Its main strengths are interoperability and tooling: HTTP is universally available, easy to debug, and integrates naturally with standard security mechanisms. The cost is higher per-message overhead due to verbose headers and text serialization, and a synchronous round trip that can become a bottleneck under high-frequency traffic. Reliability is end-to-end and typically depends on client retry logic, since the protocol does not provide broker-side buffering.

\paragraph{gRPC.}
gRPC provides typed remote procedure calls using Protocol Buffers over HTTP/2 and supports unary requests as well as streaming semantics~\cite{grpc_docs}. Binary serialization and connection multiplexing generally reduce message size and latency relative to JSON/HTTP, which is beneficial for frequent service-to-service messaging. In exchange, gRPC introduces schema and code-generation tooling and is less universally accessible than plain HTTP, which can complicate heterogeneous deployments.

\paragraph{MQTT.}
MQTT is a lightweight publish--subscribe protocol in which agents communicate via a broker and topic-based addressing~\cite{mqtt_org,mqtt_v5_0}. Persistent connections and small frame overhead make MQTT well suited for frequent small messages and large numbers of connected clients. QoS levels expose explicit trade-offs between throughput and delivery guarantees, but operational robustness depends on broker availability and high-availability support in the selected broker implementation.

\paragraph{Apache Kafka.}
Apache Kafka is a distributed event streaming platform built around partitioned, replicated logs and a producer/consumer programming model~\cite{apache_kafka_docs,kreps2014_benchmarking_kafka}. Kafka is optimized for high-throughput ingestion, durable storage, and replayable streams, enabling loosely coupled pipelines and buffering between producers and consumers. These advantages come with higher operational complexity and an interaction model that favors continuous streams over low-latency request/response; small, latency-sensitive exchanges can incur additional end-to-end delay due to broker mediation and acknowledgement policies.

\subsubsection{Benchmark design}

\paragraph{Implementation overview.}
Protocol adapters and transport abstractions are implemented in \texttt{src/benchmarks/communication/}, while scenario definitions, orchestration, and analysis utilities are provided in \texttt{src/benchmarks/communication\_benchmarks/} and \texttt{src/benchmarks/benchmark\_runner.py}. The extensive experiment matrix used for the reported results is specified in \texttt{src/benchmarks/benchmark\_configs/all\_benchmarks\_concurrency.yml}. Raw outputs (JSON/CSV summaries and plots) are stored under \texttt{src/results/}; figures are included in this section via the \texttt{results/...} paths.

\paragraph{Architecture and real network communication.}
The communication benchmarks exercise the full network stack using real protocols and servers, in contrast to hierarchy benchmarks that simulate message passing in memory. REST is evaluated via a local Flask service (JSON over HTTP/1.1 or HTTP/2)~\cite{Fielding2000REST,RFC9110}. gRPC uses Protocol Buffers over HTTP/2 for typed RPC and streaming~\cite{gRPCConcepts,gRPCProto}. MQTT uses a Mosquitto broker to assess publish--subscribe under different QoS semantics~\cite{MQTTv5Spec}. Kafka uses an Apache Kafka cluster to measure broker-mediated event streaming under different producer acknowledgement policies~\cite{apache_kafka_docs,KafkaProducerAcks}.
The benchmark runner instantiates standardized scenarios (point--to--point, broadcast, concurrent load, sustained stress), manages agents and their mailboxes, and collects metrics via latency, throughput, reliability, and resource trackers. Agents interact through protocol--specific clients that serialize and transmit messages to services or brokers.

\paragraph{Goals and design.}
The communication benchmark suite provides apples--to--apples comparisons across REST, gRPC, MQTT, and Kafka by reusing the same high--level scenarios, normalizing configuration in a single runner, and collecting a consistent set of metrics (latency, throughput, reliability, resource usage). Results are exported to JSON/CSV for cross--protocol analysis and plotting.

\paragraph{Hardware and software environment.}
All communication benchmark results in this section were collected on a single host running Ubuntu 22.04.5 LTS (Linux kernel \texttt{6.8.0-90-generic}) with an AMD Ryzen 9 5900HX (8 physical cores / 16 threads, 0.40--4.68~GHz reported) and 30.75~GB RAM, using Python 3.10.12.

\paragraph{Message Flow Example (REST, Point--to--Point).}
The listing below shows a simplified end--to--end path for REST. The sender issues a real HTTP request; the Flask server parses, enqueues, and responds; the benchmark records the measured interval.

\begin{lstlisting}[language=Python, caption={REST point-to-point message path}]
# Benchmark starts timing
benchmark.latency_tracker.start_message_timing("msg_123")

# Sender issues a real HTTP request (JSON over HTTP)
sender.send_message(
    receiver_id="agent_1",
    message_type=MessageType.INFORM,
    content={"data": "hello"}

# Under the hood: requests.post(...) to Flask service
# Flask handler enqueues the message into receiver's mailbox (thread-safe)
# HTTP 200 OK is returned

benchmark.latency_tracker.end_message_timing("msg_123")
\end{lstlisting}

This measures network transfer, serialization and deserialization, HTTP request/response handling, queue operations, and server routing.

\paragraph{Latency Modes and the Fairness Problem.}
Protocols differ in their default synchronization points. REST and typical synchronous gRPC clients block until the server responds to the request; MQTT and Kafka often expose asynchronous publishing that returns before any subscriber receives data. Measuring only the send operation would make asynchronous publish--subscribe appear faster, since broker handoff is shorter than end--to--end delivery. To address this, the benchmark supports three modes: \texttt{send\_only}, \texttt{end\_to\_end}, and \texttt{app\_ack}. This separation aligns measurement with user intent and avoids misleading comparisons \cite{Tene2015Latency,DeanBarroso2013TailAtScale,Jain1991PerformanceAnalysis}. Table~\ref{tab:protocol-behavior-measurement} summarizes the default behavior and the resulting measurement scope for each protocol family.

\begin{table}[H]

\label{tab:protocol-behavior-measurement}

\toprule
\textbf{Protocol} & \textbf{Default Behavior} & \textbf{What Gets Measured} \\
\midrule

gRPC  & Can be sync or async             & Depends on client/server implementation \\

\paragraph{Mode 2: \texttt{end\_to\_end} (Transport Round--Trip).}
Measure from the sender’s initiation until the transport's default completion point. No explicit application--level ACK is awaited; instead, the blocking call’s completion serves as the synchronization point for REST (HTTP response), gRPC (RPC response), and Kafka (producer acknowledgments when enabled). MQTT publishing is asynchronous in this harness, so \texttt{end\_to\_end} measures publish call completion (broker handoff) rather than receiver processing; \texttt{app\_ack} is used for strict end-to-end delivery.

\paragraph{Mode 3: \texttt{app\_ack} (Application--Level Round Trip).}
Extends \texttt{end\_to\_end} by measuring a full round trip including an application--level ACK sent by the receiver. Each scenario implements a lightweight ACK loop using a dedicated ACK message type so the sender stops timing only after receiving matching ACK(s). Figure~\ref{fig:timing-boundaries-ack} illustrates the timing boundaries used in this mode. This provides the strictest, protocol--agnostic comparison of end--to--end behavior when feasible.

\paragraph{Benchmark Scenarios.}
\paragraph{(1) \texttt{point\_to\_point\_latency}.}
Goal: measure basic message delivery speed between two agents over a chosen topology. Typical configuration uses 50 sequential messages in simple mode and 100 in extensive mode. Report the mean, percentiles (p95, p99), minimum, maximum, and success rate \cite{DeanBarroso2013TailAtScale,Jain1991PerformanceAnalysis}.

\paragraph{(2) \texttt{broadcast\_throughput}.}
Goal: evaluate one--to--many performance. One sender broadcasts to $N$ receivers (for example 30--50 broadcasts). MQTT and Kafka implement true publish--subscribe, which reduces sender work relative to per--receiver loops \cite{MQTTv5Spec,apache_kafka_docs}.
Note: the unified runner skips \texttt{broadcast\_throughput} in \texttt{end\_to\_end} mode, because protocol-level broadcast completion points are not directly comparable; broadcast comparisons are reported for \texttt{send\_only} and \texttt{app\_ack}.
The harness inserts a fixed 20~ms pause between broadcasts to avoid burst saturation and keep receiver queues stable; this caps the sender at roughly 50 broadcasts/s before protocol overhead.

\begin{lstlisting}[language=Python, caption={Broadcast loop with timing}]
for i in range(message_count):
    benchmark.throughput_tracker.record_message()
    benchmark.latency_tracker.start_message_timing(message_id)
    result = sender.broadcast_message(content)
    benchmark.latency_tracker.end_message_timing(message_id)
    time.sleep(0.02)   # 20 ms between broadcasts (caps ~50 broadcasts/s)
\end{lstlisting}

\paragraph{(3) \texttt{concurrent\_messaging}.}
Goal: assess behavior under concurrent load with $N$ agents sending to random peers. Simple runs use $N=4$ with 100 messages per agent; extensive runs sweep $N \in \{5,10,15,20\}$ with the same per-agent message count. A separate concurrency matrix (senders 1, 4, 8, 16 in the extensive configuration) characterizes scaling under contention.

\begin{lstlisting}[language=Python, caption={Concurrent messaging workload}]
def agent_messaging_task(agent, agent_index):
    for i in range(messages_per_agent):
        target = random.choice(other_agents)
        benchmark.latency_tracker.start_message_timing(message_id)
        agent.send_message(target.id, MessageType.INFORM, content)
        benchmark.latency_tracker.end_message_timing(message_id)

threads = [Thread(target=agent_messaging_task, args=(agent, i))
           for i, agent in enumerate(agents)]
for t in threads: t.start()
for t in threads: t.join()
\end{lstlisting}

\paragraph{Metrics and Instrumentation.}
\paragraph{Latency tracker.}
A monotonic, high-resolution clock (\texttt{time.perf\_counter}) is used for start/stop timing. Completed latencies feed percentile summaries for reporting \cite{PythonTime,DeanBarroso2013TailAtScale,Tene2015Latency}.

\begin{lstlisting}[language=Python, caption={Latency tracker}]
class LatencyTracker:
    def __init__(self):
        self.pending_messages = {}     # msg_id -> start_time
        self.completed_latencies = []

    def start_message_timing(self, message_id: str):
        self.pending_messages[message_id] = time.perf_counter()

    def end_message_timing(self, message_id: str) -> float:
        start_time = self.pending_messages.pop(message_id)
        latency = time.perf_counter() - start_time
        self.completed_latencies.append(latency)
        return latency
\end{lstlisting}

\paragraph{Throughput tracker.}
A sliding window over timestamps yields real--time throughput in messages per second \cite{Jain1991PerformanceAnalysis}.

\begin{lstlisting}[language=Python, caption={Throughput tracker}]
class ThroughputTracker:
    def __init__(self, window_size=1.0):   # seconds
        self.message_timestamps = deque()
        self.window_size = window_size
    def record_message(self):
        current_time = time.perf_counter()
        self.message_timestamps.append(current_time)
        cutoff = current_time - self.window_size
        while self.message_timestamps and self.message_timestamps[0] < cutoff:
            self.message_timestamps.popleft()
    def get_current_throughput(self) -> float:
        return len(self.message_timestamps) / self.window_size
\end{lstlisting}

\paragraph{Resource monitor.}
CPU and memory samples are collected periodically using \texttt{psutil} in a background thread \cite{psutil}.

\begin{lstlisting}[language=Python, caption={Resource monitor}]
class ResourceMonitor:
    def __init__(self):
        self.cpu_samples, self.memory_samples = [], []
        self.monitoring = True
    def _monitor_loop(self):
        p = psutil.Process()
        while self.monitoring:
            self.cpu_samples.append(psutil.cpu_percent())
            self.memory_samples.append(p.memory_info().rss)
            time.sleep(0.1)   # 100 ms sampling
    def start_monitoring(self):
        Thread(target=self._monitor_loop, daemon=True).start()
\end{lstlisting}

\paragraph{Topology Patterns.}
Scenarios can run over different communication topologies. Built--in patterns include \texttt{fully\_connected}, \texttt{star}, \texttt{ring}, and \texttt{chain}. The setup phase builds the topology and installs directed links (sender $\rightarrow$ receiver) so that each protocol exercises the same overlay.

\paragraph{Metrics Collected.}
Across scenarios the harness records a consistent set of metrics: latency (avg, min, max, std, p50, p95, p99, p99.9, jitter; sampled latencies for plots), throughput (average and 1~s--window peak; per--second history), reliability (success rate, delivery/timeouts; ordering violations and duplicates when tracked), resources (process CPU\% and RSS), and test metadata (agent count, total messages, duration, topology density, payload size, latency mode).

\paragraph{Runner Orchestration and Variants.}
The unified runner ensures protocol environments are ready (e.g., MQTT/Kafka brokers) before measurements, instantiates protocol--specific scenario factories, iterates configured protocol variants, and optionally sweeps topologies in extensive mode. Protocol variants map to common settings: REST \texttt{http1}/\texttt{http2}; gRPC \texttt{unary}/\texttt{streaming}; MQTT \texttt{qos0}/\texttt{qos1}/\texttt{qos2}; Kafka \texttt{acks0}/\texttt{acks1}/\texttt{acksall} (note: \texttt{acks0} is skipped in \texttt{app\_ack} mode). For \texttt{concurrent\_messaging}, a concurrency matrix can vary the number of concurrent senders per variant.

\paragraph{Experiment matrix.}
Benchmarks are executed with a unified runner that sweeps protocol variants (Table~\ref{tab:comm-protocol-options}), scenarios (Table~\ref{tab:benchmark-configs}), and agent counts in extensive mode. In the extensive configuration used for these results (YAML: \texttt{src/benchmarks/benchmark\_configs/all\_benchmarks\_concurrency.yml}), agent counts are swept over $N \in \{5,10,15,20\}$ on a fully connected topology. The runner additionally performs a topology sweep over \texttt{fully\_connected}, \texttt{star}, \texttt{ring}, and \texttt{chain} for \texttt{concurrent\_messaging} at $N=20$, and a concurrency matrix for \texttt{concurrent\_messaging} that varies the number of concurrent senders (senders $\in \{1,4,8,16\}$).

\paragraph{Trials and repetition.}
Each configuration point (protocol $\times$ variant $\times$ scenario $\times$ agent count $\times$ latency mode, plus optional topology/concurrency sweeps) is repeated for $K$ independent trials. In our benchmark configuration we use $K=5$ trials and 0 warm-up operations per trial; the runner exports bootstrap 95\% confidence intervals (CI) for the per-trial means to JSON/CSV.

\paragraph{Outputs.}
Simple runs use small message counts and a single agent count for quick sanity checks; extensive runs sweep agent counts and topologies, and may include a concurrency matrix. Each run writes JSON and CSV to an output directory, including full per--protocol results and cross--protocol summaries (latency, throughput, and resource usage CSVs).

\paragraph{Discussion.}
As expected, \texttt{app\_ack} yields higher measured latency and lower throughput than transport-only modes because receiver processing and the application-level acknowledgment loop are included in the timing boundaries. In these experiments, MQTT achieves the highest throughput across modes with consistently low latencies, while gRPC (unary) provides low-latency RPC communication at high throughput. REST exhibits substantially higher latencies under concurrent load, reflecting the overhead of HTTP request/response handling and text-based serialization. Kafka shows higher end-to-end costs in \texttt{app\_ack} due to broker mediation and acknowledgement semantics; \texttt{acks=all} improves both throughput and success rate relative to \texttt{acks=1} in this configuration, but remains below MQTT and gRPC for latency-sensitive communication.

\subsection{Hierarchy Benchmarks}
\label{sec:hierarchy-benchmarks}

Equally critical to an MAS architecture is how the agents are organized and how control or decision making is distributed. While the communication benchmarks measure transport-layer performance using real protocols and servers, the hierarchy benchmarks isolate coordination logic by simulating organizational strategies in memory. The benchmark suite is implemented in \texttt{src/benchmarks/hierarchy\_benchmarks/} and compares Tree, Peer-to-Peer (P2P), and Hybrid structures.

\subsubsection{Hierarchy Options (Tree, Peer-to-Peer, Hybrid)}
Inter-agent coordination can be organized around explicit managers, purely decentralized negotiation, or a mix of both. Table~\ref{tab:hierarchy-options} summarizes the three options evaluated.

\begin{table}[H]
\centering
\caption{Hierarchy strategies evaluated in the benchmark suite}
\label{tab:hierarchy-options}
\begin{tabular}{l|l|l|l}
\toprule
\textbf{Strategy} & \textbf{Structure} & \textbf{Coordination pattern} & \textbf{Expected trade-offs} \\
\midrule
Tree         & Manager--worker hierarchy (depth $d$) & Delegation + aggregation & Low overhead; bottlenecks; leader sensitivity \\
Peer-to-peer & Flat graph                             & Proposals / gossip       & Robust; higher coordination cost \\
Hybrid       & Hierarchical backbone + peer links      & Delegation + local peer exchanges & Balance between control and flexibility \\
\bottomrule
\end{tabular}
\end{table}

\paragraph{Tree.}
Agents are arranged in a manager--worker hierarchy. Managers allocate tasks, collect status, and reassign work when necessary~\cite{Moore2025Taxonomy,VanSteenTanenbaum2017DistributedSystems}. This structure can reduce coordination overhead via fan-out/fan-in, but introduces bottlenecks and sensitivity to manager failures.

\paragraph{Peer-to-peer (P2P).}
All agents are equal and coordinate via distributed proposals and agreement. This improves robustness (no fixed leader) but can increase coordination complexity and overhead as the number of agents grows~\cite{Shehory1998Architectural,Eugster2003PubSubSurvey,Kempe2003Gossip}.

\paragraph{Hybrid.}
A hierarchical backbone handles global delegation, while additional peer links support local coordination to bypass bottlenecks. Such mixed organizational designs are commonly used to balance efficiency and adaptability~\cite{Dignum2004AgentOrganizations,Wooldridge2009MAS}.

\subsubsection{Hierarchy Benchmark Design}
\paragraph{Purpose and scope.}
The hierarchy benchmarks compare organizational strategies on coordination-heavy workloads without transport noise. They measure task success, efficiency, and coordination overhead for Tree, P2P, and Hybrid strategies across multiple team sizes and environments~\cite{Wooldridge2009MAS,VanSteenTanenbaum2017DistributedSystems,Dignum2004AgentOrganizations}.

\paragraph{Execution model (simulated coordination records).}
Agents are in-memory state objects. Coordination effects are applied through direct state updates on agent records; there is no serialization or network transmission. To avoid confusion with real network messages, the benchmark counts synthetic \emph{coordination records} created for bookkeeping only. Table~\ref{tab:msg-semantics} summarizes the contrast with the communication benchmarks.

\begin{table}[H]
\centering
\caption{Simulated coordination record semantics in hierarchy benchmarks}
\label{tab:msg-semantics}
\begin{tabular}{l|l|l}
\toprule
\textbf{Aspect} & \textbf{Hierarchy} & \textbf{Communication} \\
\midrule
Creation  & Python object            & Serialized (JSON or Protobuf) \\
Sending   & Append to in-memory list & Network transmission          \\
Delivery  & Not modeled              & Real delivery to mailbox      \\
Receipt   & Not modeled              & Retrieved and processed       \\
Purpose   & Counting/proxy overhead  & Actual coordination           \\
Latency   & No network delay         & Real network delay            \\
\bottomrule
\end{tabular}
\end{table}

\paragraph{Environments.}
Benchmarks are run on three environments defined in \texttt{hierarchy\_environments.py}: \texttt{resource\_allocation} (tasks compete for limited resources), \texttt{task\_distribution} (assign and complete independent tasks), and \texttt{collaborative} (tasks have dependencies and require ordered progress).

\paragraph{Metrics.}
The suite reports task effectiveness (success rate, normalized return), efficiency (makespan in steps-to-success, action efficiency), hierarchy overhead (manager utilization, delegation success, preemption rate), and coordination footprint (coordination records per episode and estimated bytes per step). Coordination latency is measured as wall-clock time between delegation and the first observed worker progress (monotonic timer), and is therefore a proxy for coordination overhead rather than transport delay.

\paragraph{Benchmark configuration and statistics.}
Unless stated otherwise, each configuration point (strategy $\times$ environment $\times$ team size) runs 10 episodes with 5 tasks per episode. Team size is swept over $N \in \{3,5,8,12\}$ agents. Episodes terminate on success or after a fixed step budget (default: 100 steps). Coordination record ``bytes'' are estimated using a fixed envelope size per record (100 bytes) for interpretability; this is not network traffic.

\paragraph{Environment success criteria.}
To improve interpretability of success/failure conditions, two environments apply explicit thresholds. In \texttt{resource\_allocation}, resources are allocated when a task enters \texttt{in\_progress} and released when it leaves that state (completion or preemption). An episode succeeds if at least 90\% of tasks complete and no resource exceeds its capacity beyond a small tolerance (default 5\%). In \texttt{collaborative}, tasks that start out of dependency order are re-queued (freeing their worker) instead of causing irrevocable failure. Success requires a high completion fraction (default 90\%) and tolerates a bounded rate of dependency violations (default 40\% of tasks may be re-queued).

\paragraph{Ablation studies.}
In addition to cross-strategy comparisons, we run single-parameter sweeps on the Tree strategy in the \texttt{task\_distribution} environment (8 agents, 5 episodes). Baseline settings are hierarchy depth 2, planning frequency 1, and no coordination record limit (None).

\paragraph{Limitations.}
These benchmarks do not model network effects, serialization overhead, or delivery failures. They complement the communication benchmarks, which measure those aspects explicitly. Use both to separate organizational effects from protocol effects~\cite{VanSteenTanenbaum2017DistributedSystems,DeanBarroso2013TailAtScale}.

\paragraph{Example implementations (simplified).}
The benchmark suite implements these strategies in Python; Listings~\ref{lst:tree-simplified}--\ref{lst:hybrid-simplified} provide simplified sketches of the core control loops.

\begin{lstlisting}[language=Python, caption={Tree: manager delegation and worker progress}, label={lst:tree-simplified}]
class AgentState:
    def __init__(self, agent_id, role, supervisor=None):
        self.agent_id = agent_id
        self.role = role              # "manager" or "worker"
        self.supervisor = supervisor  # None for root
        self.subordinates = set()
        self.current_task = None
        self.progress = 0.0

class TreeHierarchy:
    def initialize(self, num_agents):
        self.agents = {}
        self.agents["agent_0"] = AgentState("agent_0", "manager")
        for i in range(1, num_agents):
            wid = f"agent_{i}"
            self.agents[wid] = AgentState(wid, "worker", supervisor="agent_0")
            self.agents["agent_0"].subordinates.add(wid)

    def allocate_task(self, task):
        # direct state update (no real network message)
        idle = [a for a in self.agents.values() if a.role == "worker" and a.current_task is None]
        if not idle: return False
        worker = random.choice(idle)
        worker.current_task = task.task_id
        # optional: record a synthetic "delegation" coordination record for counting
        return True

    def step(self):
        # workers make simulated progress
        for a in self.agents.values():
            if a.role == "worker" and a.current_task:
                a.progress += 0.05  # step size encodes "capacity"
\end{lstlisting}

\begin{lstlisting}[language=Python, caption={P2P: simple proposal broadcast and local decision}, label={lst:p2p-simplified}]
class PeerToPeerHierarchy:
    def initialize(self, num_agents):
        self.agents = {f"agent_{i}": AgentState(f"agent_{i}", "peer") for i in range(num_agents)}
        self.peers = {i: set(self.agents) - {i} for i in self.agents}

    def allocate_task(self, task):
        # pick candidate, broadcast proposal to peers (synthetic coordination records)
        candidate = random.choice(list(self.agents))
        # tally "votes" locally (no network)
        # accept if no conflict detected
        self.agents[candidate].current_task = task.task_id
        return True

    def step(self):
        for a in self.agents.values():
            if a.current_task:
                a.progress += 0.04
\end{lstlisting}

\begin{lstlisting}[language=Python, caption={Hybrid: hierarchical delegation with peer side-channels}, label={lst:hybrid-simplified}]
class HybridHierarchy(TreeHierarchy):
    def step(self):
        # same as tree for global delegation
        super().step()
        # extra peer "nudges" for local coordination (synthetic coordination records)
        # e.g., siblings swap tasks to reduce imbalance
\end{lstlisting}
